## Props vs state at a glance

<AnnotatedCode
  title="ProductCard.tsx"
  language="tsx"
  code={`interface ProductCardProps {
  name: string;
  price: number;
  currency?: string;
}

function ProductCard({ name, price, currency = 'USD' }: ProductCardProps) {
  // State: local, mutable — the component controls this
  const [quantity, setQuantity] = useState(1);

  // Derived value: compute from props, no state needed
  const total = price * quantity;

  return (
    <div>
      <h3>{name}</h3>
      <p>{currency} {price.toFixed(2)} × {quantity} = {total.toFixed(2)}</p>
      <button onClick={() => setQuantity(q => q + 1)}>Add one</button>
    </div>
  );
}`}
  annotations={[
    { line: 1, label: 'Props interface', explanation: 'name and price come from the parent — this component cannot change them. currency has a default, but it is still read-only.' },
    { line: 9, label: 'Local state', explanation: 'quantity is local — only this component manages it. The parent knows nothing about it. That is intentional: keep state as close to where it is used as possible.' },
    { line: 12, label: 'Derived value', explanation: 'total is computed from price * quantity. It does not need its own state — it updates automatically whenever price or quantity changes.' },
  ]}
/>

<AnalogyBridge
  concept="Props vs state"
  realWorld="A restaurant menu vs. your order"
  analogy="Props are the menu — set by the kitchen, readable by the customer, unchangeable at the table. State is your current order — you can modify it as much as you like, and it only affects your table."
/>

<DangerZone
  title="Do not store derived data in state"
  wrong={`function PriceDisplay({ price, quantity }) {\n  const [total, setTotal] = useState(price * quantity);\n\n  // Bug: total is stale whenever price or quantity changes\n  return <p>Total: {total}</p>;\n}`}
  right={`function PriceDisplay({ price, quantity }) {\n  // Compute inline — always fresh, zero extra state\n  const total = price * quantity;\n  return <p>Total: {total}</p>;\n}`}
  explanation="Storing a value in state that can be computed from existing props or state introduces a second source of truth. These drift out of sync and cause subtle bugs. Compute it inline instead — React re-renders fast enough that this is never a performance problem at this scale."
/>

<DeepDive title="When to use props vs state">
  Use **props** when: the value is decided by the parent, it never changes from the child's perspective, or multiple components need the same data. Use **state** when: the value can change over time due to user interaction, it belongs to exactly one component, and no other component needs to read it. When two sibling components need the same value, lift it to their common parent as state and pass it down as props.
</DeepDive>
