# Your First Web Part

<ConceptCard
  title="Scaffold with yo, serve with heft start"

## Step 1 — Scaffold the project

<AnnotatedCode
  title="Terminal — scaffold a new SPFx project"
  language="bash"
  code={`mkdir my-first-webpart
cd my-first-webpart
yo @microsoft/sharepoint

# The generator will ask:
# - Solution name: my-first-webpart
# - Target: SharePoint Online only
# - Component type: Web part
# - Web part name: HelloWorld
# - Framework: React`}
  annotations={[
    { line: 3, label: 'yo @microsoft/sharepoint', explanation: 'This runs the Yeoman generator. It asks a series of questions then creates the full project structure: config/, src/, and all required JSON files.' },
    { line: 9, label: 'Framework: React', explanation: 'Always choose React unless you have a specific reason not to. The React template generates a TypeScript web part class that bootstraps a React component via ReactDom.render().' },
  ]}
/>

## Step 2 — Start the dev server

<AnnotatedCode
  title="Terminal — start local dev server"
  language="bash"
  code={`cd my-first-webpart
heft start

# Output: Local server running at https://localhost:4321/
# The server watches for changes and rebuilds automatically`}
  annotations={[
    { line: 2, label: 'heft start', explanation: 'heft start replaces the old gulp serve command. It starts a local HTTPS server on port 4321 that hosts your web part bundle. SharePoint will load the bundle from here during debug.' },
  ]}
/>

## Step 3 — Load in SharePoint

The hosted workbench (`/_layouts/15/workbench.aspx`) is deprecated. Instead, use a real SharePoint page with debug query parameters:

<AnnotatedCode
  title="Debug URL format"
  language="text"
  code={`https://yourtenant.sharepoint.com/sites/yoursite/SitePages/Home.aspx
  ?debug=true
  &noredir=true
  &debugManifestsFile=https://localhost:4321/temp/manifests.js`}
  annotations={[
    { line: 2, label: 'debug=true', explanation: "Tells SharePoint to load web parts in debug mode — it will look for local manifests instead of only using deployed bundles." },
    { line: 3, label: 'noredir=true', explanation: "Prevents SharePoint from redirecting you away from the page if it detects you're in debug mode." },
    { line: 4, label: 'debugManifestsFile', explanation: 'Points to your local Heft dev server. This is where SharePoint loads your web part bundle from during development.' },
  ]}
/>

<DeepDive title="What lives in manifest.json">
  Every SPFx web part has a `<webpartname>.manifest.json` file that SharePoint reads before loading your bundle. It declares the component ID (a GUID), the display name shown in the web part toolbox, the preconfigured entries (default property values), and the bundle path. The GUID must be unique and stable — changing it after deployment breaks existing page references.
</DeepDive>
